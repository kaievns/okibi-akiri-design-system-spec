the problem

Most themes shout, all the time

Traditional highlighting colours the grammar — every keyword, every type, every bracket, lit all day, whether you're reading or not. Every element lit dims every other. Salience spent everywhere is salience spent nowhere.

Same code. On the right, calls carry the one strong hue, keywords and brackets recede into the spine, and true keeps its red because a decision is worth more than a digit. Nothing shouts — so the little that speaks, means.

the doctrine, in one picture

Salience is a budget

A healthy view spends almost nothing at rest. Structure rides luminance and weight — high-bandwidth, colour-blind-safe — and hue is rationed to a few meanings. The ember appears once or twice per screen: focus and action, never decoration.

ground · ~85% structure
ground — surfaces & body text structure — borders, labels, gutters meaning — the quiet pigments, ~3% the ember — <1%
One ember, kept sacredCursor, active tab, primary action. Never a token, never a link, never decoration.
Redundant where it countsAnything that must be told apart carries shape or weight beside hue — it works in monochrome.
Transitions are cutsNothing fades, pulses, or slides. State changes on intentional boundaries, never on drift.
the novel part

Highlighting that follows intent

Code colouring is three cumulative bands gated by what you're doing. Reading a file is not editing a line — so the theme doesn't pretend it is. Click through the phases:

// module state, far above… const registry = new Map(); let hits = 0; export async function resolve(name, scale = 1) {focused scope let theme = registry.get(name); if (theme) { hits += 1; return theme; } theme = await load(name); registry.set(name, theme); return theme; }
