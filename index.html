Okibi.
おきび — banked embers, the coal that holds the fire

A colour system for code and the interfaces around it. Built on one principle — that attention is the only budget worth spending — and dressed in the restraint of charcoal, ash, and a single ember. This is the canonical reference: the philosophy, the palette, and enough of the syntax model to re-implement it anywhere.

00 · ri

Philosophy

り · the reasoning

Most syntax themes inherit their colours from language authors — people who spent years obsessing over keywords and type systems, and who naturally want those highlighted. Okibi steps back and asks a different question: when you look at code, what do you actually need to see?

The answer is rarely the keywords. Every language is Turing-complete; the if, the func, the const are the most boring, most repeated, most predictable tokens on the screen. What you reach for is what the program does — its data and its transformations. The variables. The calls. The values flowing through. Okibi is built to put those first and let the scaffolding recede.

So colour here is not decoration and not a taxonomy of grammar. It is an attention budget. Every hue spent is attention drawn; spend it only where it earns focus. The result is a surface that reads almost like prose — calm, mostly monochrome — with colour arriving exactly where meaning lives.

Coal & ash, one ember · 燠火

The name is okibi — banked embers, the glowing coals that hold a fire through the night. The visual world follows: a deep charcoal ground, a long ash-grey ramp for everything structural, and a single warm ember that is the one point of heat in the whole system. The ember is sacred — it marks focus and action only (the cursor, the active element, the primary button) and never colours a token. Where there is fire, there is intent.

Glow in the dark, not in your face

The accent hues are drawn from traditional Japanese pigment — but read through an action-anime night rather than a tea ceremony. They should feel like light sources in the dark: fresh, a little electric, confident. Tasteful, never loud. Saturation is earned, not sprayed; a colour that appears everywhere stops meaning anything. Restraint is the whole point — when nearly everything is ash, the few colours that remain carry real weight.

Two grounds, one discipline

Okibi (charcoal) and Akari 灯り (paper light) are not two themes — they are one system seen by two lights. The same roles, the same relationships, the same restraint; only the ground inverts. Dark leans faintly cool (the blue of dusk on coal); light leans faintly warm (washi paper). A design that holds in one holds in the other.

01 · soku

Principles

そく · the rules to build by
01
焦

Attention is the budget

Colour draws the eye. Spend it only on what deserves focus — data and transformations first, scaffolding last. If everything is highlighted, nothing is.

02
灰

Default to ash

~85% of any surface is the neutral ramp. Structure — panels, borders, keywords, types — recedes into grey. Neutral is the canvas, not a fallback.

03
燠

One ember, kept sacred

A single warm accent marks focus and primary action. It never colours a syntax token and rarely appears more than once or twice per screen.

04
少

Few hues, far apart

Three or four meaning-colours, each on a distinct part of the wheel and at one shared intensity. No crowding, no near-duplicates competing.

05
意

Colour carries meaning

A hue always means the same thing across the whole system — chrome and code alike. Green is a string and a success; blue is a call and a link. One source of truth.

06
静

Calm over clever

No italics-as-decoration, no bold keywords shouting, no rainbow brackets. The surface should feel quiet enough to read for hours.

02 · ji

Two grounds

じ · the field beneath everything

One system, two lights. Pick the ground for the room; everything else follows the same rules.

Okibi 燠火
Banked-ember charcoal. A faintly cool blue-black (hue 250) so the darks tie to the cool accents. For night, for focus, for long sessions.
ground #07080a · text #e1e3e5
Akari 灯り
Washi paper light. A faintly warm off-white (hue 80) — never clinical white. The same discipline inverted, for daylight and print.
ground #e1ddd8 · text #13110e
燠

The grounds are mirror images, not independent palettes. Author once in OKLCH; derive the second ground by inverting lightness and nudging the neutral hue (cool→warm). Accent hues keep their hue and chroma, dropping lightness for the light ground so they hold contrast on paper.

Accent & states

おき · the ember & the signals

One warm ember owns focus and action. A tight set of state colours — drawn from the same hues the code uses — carries success, warning, error, and information. Chrome and code share one source of colour.

The ember is orange; the error is red. Keep them apart. Because focus and alarm sit near each other on the warm side, the ember (hue 38, orange) and error (hue 24, true red) are deliberately spaced so a cursor never reads as a problem. Warning (yellow-gold, hue 84) sits clear of both.

05 · kō

The syntax model

こう · how code is coloured

Syntax highlighting in Okibi is an attention hierarchy, not a grammar map. Tokens are ranked by how much they tell you about what the program does, and colour is assigned in that order. Read the ladder top-to-bottom: the eye should land on the same order.

1Variables · dataPlain foreground (n11). The brightest thing on screen — what you read most.
2Calls · functionsSora blue. The verbs — where transformation happens. The strongest hue.
3StringsWakaba green. Literal text — distinct, easy to scan, never confused with code.
4Numbers · constantsKurenai red. Fixed values, booleans, enum members — data that doesn't move.
5Members · propertiesPale sky. Data you reach into — a gentle tier, present but quieter than calls.
6TypesNear-neutral ash. Structural nouns — barely tinted, they recede.
7Keywords · storageSuzu grey. The scaffolding — present but the most recessive coloured role.
8CommentsMuted grey, italic. Read when sought, invisible when not.
9PunctuationDimmest. Brackets, commas, semicolons — the mortar between bricks.

The model in code

Every coloured role at its assigned weight — note how the calls and data carry the line, while pub, fn, struct and the type names sink into the grey.

Types and keywords are deliberately the quietest coloured roles. This is the sharpest break from convention — most themes make types vivid. Okibi treats them as structural nouns that you infer from shape and context, freeing the strong hues for the data and the calls that tell you what is happening.

Scope mapping

わり · roles to grammar scopes

To re-implement Okibi for any grammar — TextMate, Tree-sitter, LSP semantic tokens, an editor's own scheme — map these grammar scopes onto the nine roles. The role is the contract; the exact scope names vary by tool.

割

Heuristics, where scopes fall short. A faithful Okibi needs only a few rules beyond raw scopes: an identifier followed by ( is a call; an identifier after . is a member; an identifier before : in a declaration is a field; PascalCase and known primitives are types; ALL_CAPS is a constant. These keep the model consistent across languages whose grammars disagree.

作
08 · saku

Building with Okibi

さく · across every surface

The same system carries any surface — a web app, a native window, a terminal UI. The medium changes the resolution of colour available, never the rules. Map the ground to the spine, spend the ember once, and let the meaning hues do the rest.

Universal rules

◇ do

  • Build every surface, border and label from the spine (n0–n11).
  • Reserve the ember for focus and the single primary action per view.
  • Keep one hue = one meaning across chrome and code.
  • Step elevation by lightness (n2 → n3 → n4), not by drop shadows alone.
  • Hold text contrast at ≥ 4.5:1 for body, ≥ 3:1 for large / dim labels.

◇ don't

  • Tint large fills with accent colour — accents are for marks, not areas.
  • Use more than one ember moment in a single view.
  • Introduce a hue that isn't already in the system.
  • Make types or keywords loud — they recede, always.
  • Rely on colour alone for state; pair it with icon, weight, or shape.

Per surface

網

Web & application UI

full OKLCH

Full colour fidelity — author the tokens directly as CSS custom properties. Body on n2, raised cards on n3–n4, borders on n6. Primary buttons take the ember fill with ember-fg text; secondary actions are n4 with an n6 border. Links use the call (sora) hue, never the ember. Selection and focus rings use ember-soft / ember-line.

page → n0surface → n2card → n3/n4border → n6primary → emberlink → call
窓

Desktop & native

window chrome

Title bar and sidebars on n1; content on n2. Use the spine for the whole window chrome so the app sits inside the OS quietly. The active tab / focused pane gets a thin ember edge (a 2px inset border), not a fill. Window-control and traffic-light affordances may borrow the state colours (error/warn/success) but keep them small.

titlebar → n1sidebar → n1content → n2active edge → embergit state → success/warn
端

TUI & terminal apps

truecolor

If the terminal supports 24-bit colour, use the hex values directly — the model maps cleanly. Panel backgrounds on n1/n2, borders on n6, dim chrome text on n8–n9. Selected rows use ember-soft; the focused pane border uses the ember. Status/mode lines stay on the spine with a single ember marker — resist the urge to paint the whole bar.

panel → n1border → n6dim → n8selection → ember-softfocus border → ember

作

One test for any surface: squint at it. You should see a calm charcoal (or paper) field, the text legible in near-monochrome, and — if you've spent it well — exactly one warm spark where the work is happening. If you see a constellation of colours competing, spend less.
